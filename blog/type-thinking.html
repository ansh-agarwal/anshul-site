Type Thinking

Designing so the wrong thing can't be written down

Most bugs aren't logic errors. They're states that should never have existed in the first place — an order that's both shipped and cancelled, a price with no currency, a "custom" payment option carrying a fixed amount. We then spend our careers writing defensive checks to catch states we accidentally made possible. Type thinking flips the order of operations: instead of allowing everything and validating after, you make the illegal states unrepresentable, so the validation is structural and the compiler is on your side.

The core move: model the domain, then let the shape forbid mistakes

Consider a payment that can be paid in cash, in points, or split between them. The lazy model is a bag of optional fields:

// everything optional → every combination "possible"
{ cash?: number, points?: number, isCustom?: boolean, min?: number, max?: number }

Now cash and points can both be null. isCustom can be true with a fixed cash. Every consumer must re-check the same invariants, forever. The typed version names the real choices and closes the door:

type PaymentOption =
  | { kind: "FIXED";  cash: Money; burn: Money }
  | { kind: "CUSTOM"; lever: "burn"; range: { min: Money; max: Money; step: number } }

A discriminated union. There is no way to construct a CUSTOM option with a fixed amount — the field simply isn't there. The check that used to live in twelve call-sites now lives once, in the type. This is the same reasoning that turned a messy offers API into a clean contract: the structure carries the rules.

Make impossible states impossible

A few patterns do most of the work:

  • Sum types over flag soup. "One of these shapes" beats "many optional fields that happen to combine."
  • Newtypes over primitives. Money, UserId, Points — not number, number, number. A function that takes (Money, Points) can't be called with the arguments swapped.
  • Parse, don't validate. Convert untrusted input into a precise type once, at the boundary. Past that line, the rest of the system trusts the type instead of re-checking strings.
  • Non-empty where empty is a bug. If a list must have at least one element, say so in the type rather than asserting at runtime.
The goal isn't more types. It's fewer reachable states — ideally exactly the ones the business actually allows.

Why it scales (and why leaders should care)

Tests check the cases you remembered. Types check the cases you didn't. When a new engineer joins, a precise type is the cheapest, most honest documentation there is — it tells them what's possible without a meeting. And when requirements change, you change the type and the compiler hands you the exact list of places to update. That feedback loop is the difference between a codebase that ossifies and one that stays malleable as it grows.

Type thinking isn't about a particular language or about ceremony. It's a design discipline: spend your effort up front making the wrong thing impossible, so you spend far less of it later chasing the wrong thing down.
