About

I build and lead engineering for systems that have to hold up — APIs, platforms, and the messy real-world flows behind them. My instinct is to start from the problem, not the framework: model the domain honestly, make the hard cases representable rather than patched-over, and let good structure carry the weight so the code — and the team — stays simple as it grows. I'm equally at home going deep on a type model or a data flow and stepping back to set technical direction. Most of my best work is unglamorous: turning a tangled requirement into a clean contract, and a fragile system into one people can reason about.
